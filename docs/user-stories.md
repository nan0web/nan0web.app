# User Stories: Архітектурний Рефакторинг v1.4.0

Цей документ містить 33 канонічних сценарія розробки для оновлення системної архітектури `nan0web.app` та впровадження нових механік безпеки.

## Тема 1: Міграція Базових Моделей (Універсалізація UI)

1. Як Архітектор, я хочу щоб `HeaderModel` імпортувався безпосередньо з `@nan0web/ui/domain`, аби не дублювати його в різних додатках інфраструктури.
2. Як Архітектор, я хочу щоб `FooterModel` знаходився в `@nan0web/ui/domain`, забезпечуючи єдиний стандарт корпоративних підвалів.
3. Як Архітектор, я хочу щоб `HeroModel` та `FeatureGridModel` були експортовані з `@nan0web/ui/domain`.
4. Як Розробник компонентів, я хочу щоб лінійка моделей з `extra.js` (`PriceModel`, `CommentModel`, `TestimonialModel`, `AccordionModel`, `BannerModel` тощо) стала невід'ємною частиною ядра пакету `@nan0web/ui`.
5. Як Архітектор, я хочу щоб у текці `nan0web.app/src/domain/` не залишилося жодного суто візуального (UI-специфічного) класу. Тоді `nan0web.app` залишається лише чистим раннером.

## Тема 2: Архітектурна Гігієна та Shebang

6. Як DevOps, я хочу щоб ендпойнти API (виконуваний код) не зберігались у директорії `data/`, оскільки `data/` має містити виключно декларативні формати (`.nan0`, `.yaml`, `.json`, `.md`).
7. Як DevOps, я хочу щоб для серверних хендлерів та бекенд-логіки існувала окрема, семантично вірна структура в додатку (наприклад, `src/server/api/`).
8. Як Розробник, я хочу щоб тестові специфікації (на кшталт `v1.1.0/task.spec.js`) не вимагали наявності `shebang` (`#!/usr/bin/env node`) у суто функціональних файлах `src/cli.js`.
9. Як Архітектор, я хочу щоб `shebang` розміщувався рівно в одному файлі системи (раннері), наприклад у `bin/nan0web`.

## Тема 3: Механіки PagesRouter

10. Як Архітектор, я хочу мати чітку пояснювальну документацію щодо того, як `PagesRouter` трансформує вміст декларативних даних та генерує відповідні роути при завантаженні системи.
11. Як Розробник, я хочу знати, як `PagesRouter` створює плоский словник індексів (routes) для швидкого маршрутизаційного доступу за О(1).
12. Як Користувач серверу, я хочу щоб запити на неіснуючі сторінки перехоплювались і `PagesRouter` коректно обслуговував 404-сторінку з дотриманням наявного рендерера.

## Тема 4: Безпека та DB Security (db.seal)

13. Як Секьюріті-інженер, я хочу щоб щойно конфігурації платформи і підключення до бази даних ініціалізувалися, система автоматично викликала `db.seal()`.
14. Як Секьюріті-інженер, я хочу щоб після виклику `db.seal()`, будь-які спроби модифікації ключових політик або налаштувань бази даних блокувались.
15. Як Розробник, я хочу щоб функція `db.seal()` була на 100% покрита юніт-тестом, який імітує неавторизоване втручання та переконується у недоторканості середовища.
16. Як DevOps, я хочу щоб консольний `AppRunner` при старті писав про успішне застосування `db.seal()` в лог.
17. Як Архітектор, я хочу бути повністю впевненим, що після підключення дочірніх мікрододатків, вони матимуть права працювати лише в межах заморожених конфігурацій (Read-Only для конфігів).

## Тема 5: Конфігурація Підключення БД (DB Configs)

18. Як Системний адміністратор, я хочу щоб налаштування доступу до бази даних (наприклад DSN, токени, хости) описувалися формалізованою декларативною структурою `DBConfig`.
19. Як Системний адміністратор, я хочу щоб інтеграція підключення (`connection string` або DSN) підтримувалась на рівні `NaN0WebConfig`, так щоб запуск можна було керувати одним конфігом.
20. Як Користувач CLI, якщо я ініціалізую проект з помилковим конфігом бази даних, система повинна вивести зрозуміле повідомлення про невдале підключення.
21. Як Архітектор, я хочу щоб при завершенні кожної підзадачі всі тести (`npm run test:all`) продовжували стабільно працювати без деградації попереднього функціоналу.

## Тема 6: Композиція Мікрододатків (Editor & Auth)

22. Як Адміністратор, я можу додати `editor.app` через команди `nan0web add @nan0web/editor.app` та `nan0web config`, налаштувавши, чи має компонент редактора пакуватися в кінцевий фронтенд-білд.
23. Як Редактор (Host-режим), якщо `editor.app` **не спаковано** у фронтенд, я можу вносити зміни до сайту лише при локальному SSR-запуску (git clone -> npm start) через браузер абоCLI.
24. Як Відвідувач (Клієнтський режим 1), якщо редактор запаковано у фронтенд, я можу оптимістично редагувати сторінку у браузері, залишаючи зміни лише локальними без відправки на сервер (sandbox-відображення).
25. Як Відвідувач (Клієнтський режим 2 - Wiki), якщо сервер налаштовано приймати публічні правки, мої зміни у фронтенді відправляються через API на сервер і зберігаються у `data/` глобально всіма без авторизації.
26. Як Секьюріті-інженер, я підключаю `auth.app`, щоб налаштувати доступ до API збереження `editor.app`, дозволяючи публікацію лише користувачам із відповідними правами доступу та токенами.
27. Як API (`auth.app` Middleware), при спробі неавторизованого доступу до захищених шляхів я повертаю фундаментальні статуси `401 Unauthorized` або `403 Forbidden`, не знаючи деталей візуального рендерингу.
28. Як Frontend-Клієнт (UI Adapter), я міщу вбудований механізм перехоплення: отримавши від захищеного API статус `401` або `403`, я ініціюю переадресацію користувача на сторінку логіну чи показую попап доступу.
29. Як Редактор, я маю три опції **експорту**: (a) **інкрементальний** — патч з посиланням на конкретну ревізію (HEAD), щоб інші користувачі могли коректно застосувати його навіть після нових змін; (b) **частковий** — повний вміст лише змінених документів; (c) **повний** — архів усіх документів `data/` як snapshot.
30. Як Редактор, я можу **імпортувати** патч-файл (інкрементальний або частковий), при цьому система автоматично перевіряє сумісність з поточною ревізією та при конфлікті пропонує merge або відхилення.
31. Як Редактор, перед збереженням я можу переглянути **diff** (структурну різницю між двома станами `.nan0` документа), щоб свідомо контролювати свої правки.
32. Як Система, я зберігаю історію змін через **абстрактний контракт `db.history(key)`** пакету `@nan0web/db`. Кожен DB-адаптер (db-fs, db-redis, db-browser) реалізує цей контракт по-своєму: db-fs може спиратися на git, db-redis зберігає ревізії як окремі ключі з TTL, db-browser утримує журнал у IndexedDB.
33. Як Адміністратор, у панелі редактора я маю UI для перегляду історії версій сторінки, порівняння будь-яких двох ревізій (diff-view) та відкату до обраної версії — незалежно від того, який DB-адаптер використовується.
