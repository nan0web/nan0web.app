# 🏛️ Архітектура Документації NaN0Web (Open-Source Native DDDS)

Головна філософія — **робота "з коробки" у найбільш звичному для Open Source розробників форматі** (GitHub / GitLab). 
Документація не повинна будуватись на якихось інших стандартах, ніж звичайна `nan0web` data-db. Змінюється лише точка входу і дефолтні поведінки.

---

## 1. 📂 Структура Даних: Єдиний Стандарт
Ми не пишемо специфічних умов під "документацію". Робоча директорія `docs/` поводиться як звичайна DSN.

```text
/ (корінь репозиторію)
  ├── package.json / pyproject.toml  <-- Джерело правди для UI та компонентів
  ├── README.md                      <-- Головна сторінка (fallback для index)
  ├── docs/                          <-- Звичайна база даних / папка
  │   ├── index.md
```

**Ін'єкція Локального Проєкту (Playground)**: Оскільки рушій автоматично читає поточний `package.json` і його `exports`, уся документація (Markdown) має миттєвий доступ до компонентів та доменних моделей поточного пакету. 
Розробник може просто написати в Markdown:
```html
<Button variant="primary">Клік</Button>
```
І рушій відрендерить реальний React/Lit компонент з проекту.

---

## 2. 🚀 Zero-Config Runner (`nan0web run docs`)
Запускає середовище з невеликими зручними дефолтами для розробників:

1. **In-Memory Default Layout**: Якщо в `docs/` немає файлу налаштувань макету, рушій автоматично генерує стандартний макет:
   - **Header**: Навігація.
   - **Content**: Динамічна область для відмальовування Markdown.
   - **Footer**: Ліцензія, копірайти та соцмережі з маніфесту проекту.
2. **Auto-Promo (Hero)**: На базі маніфестів екосистеми автоматично формується преміальна `Hero`-секція.

---

## 3. 🔀 Версіювання як Опція
Ми відмовляємось від нав'язливого хардкод-версіювання з коробки. Якщо розробнику не потрібні версії — їх не буде.
Щоб увімкнути селектор версій, достатньо в кореневому конфігу (`nan0.yaml` або в `index` файлі) додати:
```yaml
$versioning: true
```
Після цього з'являється `<VersionSelect>`, який читатиме доступні Git-теги і здійснюватиме зовнішній редирект на архівні субдомени (напр. `v1-1-0.nan0web.app`). У файловій системі завжди лежить лише поточна "активна" гілка (ми не смітимо дублями).

## 4. 🧠 Векторний Пошук на Клієнті (Semantic AI Search)
Звичайний `includes` пошук (чи навіть зворотний індекс FlexSearch) — це минуле століття. Ми переходимо на **Семантичний Пошук**.

**Як це працює і скільки важить (Для 1 Золотої Тисячі сторінок):**
1. 1000 документів по 1000 слів = ~1 млн слів.
2. Під час білду (SSG) або першого запуску індексатора, тексти розбиваються на "чанки" (абзаци).
3. Для кожного чанку генерується **Векторний Ембеддінг** (Embedding) розмірністю, наприклад, 384 виміри.
4. **Вага бази**: Завдяки INT8-квантизації, 1 вектор важить лише 384 байти. Навіть для 2000-3000 чанків (увесь твій 1 млн слів), вся база векторів важитиме близько **~1 МБ** (у gzip — ще менше).

**Виконання в браузері (Client-side):**
- Клєнт завантажує UI-документацію + векторну базу `search_index.bin` (~1 МБ).
- Інтерактивне вікно пошуку завантажує ультра-легку WebAssembly модель нейромережі (напр. через `@xenova/transformers.js` розміром 15-20 МБ, яка кешується браузером назавжди).
- Користувач вводить запит *"Як обгортати помилки?"*. Модель в браузері за 20мс перетворює це на вектор, порівнює по косинусній відстані з нашою базою на 1МБ і видає 100% семантично точний результат (незалежно від того, яким словом написано в статті).

Для монументальних проектів (100K+ сторінок) цей векторний пошук буде просто мігрувати у віддалений Edge / RPC Server, щоб не віддавати 100мб базу клієнту. Але для звичайних пакетів (1000 сторінок) AI-пошук повністю працюватиме в браузері автономно.
